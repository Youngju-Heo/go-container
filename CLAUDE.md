## 1. 코딩 전에 먼저 생각하라

**추측하지 마라. 혼란을 숨기지 마라. 트레이드오프를 드러내라.**

구현 전에:

- 가정을 명시적으로 밝혀라. 확신이 없으면 물어봐라.
- 해석이 여러 가지라면 모두 제시하라 - 혼자 조용히 고르지 마라.
- 더 간단한 방법이 있다면 말하라. 필요하다면 반박하라.
- 불분명한 부분이 있다면 멈춰라. 무엇이 혼란스러운지 짚어라. 질문하라.

## 2. 단순함이 먼저다

**문제를 해결하는 최소한의 코드. 추측성 코드는 없다.**

- 요청받지 않은 기능은 추가하지 마라.
- 한 번만 쓰이는 코드에 추상화는 만들지 마라.
- 요청되지 않은 "유연성"이나 "설정 가능성"은 넣지 마라.
- 일어날 수 없는 시나리오에 대한 에러 처리는 하지 마라.
- 200줄을 썼는데 50줄로 가능하다면, 다시 써라.

스스로에게 물어라: "시니어 엔지니어가 이걸 보고 과하다고 할까?" 그렇다면 단순화하라.

## 3. 외과적 변경

**꼭 필요한 부분만 건드려라. 너 자신이 만든 흔적만 정리하라.**

기존 코드를 수정할 때:

- 인접한 코드, 주석, 포맷팅을 "개선"하려 들지 마라.
- 망가지지 않은 것을 리팩터링하지 마라.
- 너라면 다르게 짰을 코드라도, 기존 스타일을 따라라.
- 무관한 데드 코드를 발견했다면 언급은 하되 삭제하지 마라.

너의 변경으로 인해 고아가 생겼다면:

- 너의 변경 때문에 쓰이지 않게 된 import/변수/함수만 제거하라.
- 요청받지 않은 한, 기존부터 있던 데드 코드는 건드리지 마라.

판단 기준: 변경된 모든 줄은 사용자의 요청과 직접 연결되어야 한다.

## 4. 목표 중심 실행

**성공 기준을 정의하라. 검증될 때까지 반복하라.**

작업을 검증 가능한 목표로 변환하라:

- "검증 추가" → "유효하지 않은 입력에 대한 테스트를 작성하고 통과시켜라"
- "버그 수정" → "버그를 재현하는 테스트를 작성하고 통과시켜라"
- "X 리팩터링" → "변경 전후로 테스트가 통과하는지 확인하라"

여러 단계로 이루어진 작업은 간단한 계획을 명시하라:

```
1. [단계] → 검증: [체크]
2. [단계] → 검증: [체크]
3. [단계] → 검증: [체크]
```

명확한 성공 기준은 독립적인 반복을 가능하게 한다. 모호한 기준("작동하게 해라")은 끊임없는 확인을 요구한다.

## 5. 문서화

** 기획/계획을 작성할때 반드시 markdown 파일로 저장하라. 모든 작업내역을 단계별로 저장하라. **

- 새로운 기획 시작시 `{feature-name}`기반으로 브랜치를 만들고 시작하라
- 기획/계획 작성시 반드시 작성된 내용을 문서(markdown 파일)로 저장하라
- 기획 작성 시 superpowers/brainstorming 스킬을 사용
  - 저장 경로·명명규칙 `docs/feature/yyyy-mm-dd-{day-sequence}-{feature-name}-design.md`
  - `{day-sequence}`는 그날 순번 2자리, `01`부터
- 개발계획 작성 시 superpowers/writing-plans 스킬을 사용
  - 저장 경로·명명규칙은 `docs/feature/yyyy-mm-dd-{day-sequence}-{feature-name}-plan.md`
  - `{day-sequence}`는, 해당 디자인과 동일한 순번
- 작업 단계별로 git에 커밋하여 이력을 저장하라
- 모든 문서와 커밋은 반드시 한국어로 작성하라.
- 코드 내부의 로그, 에러 메세지 등은 가능한 영어식 표현을 사용하라.

## 6. 개발 진행 및 스킬, 모델 규칙

**모든 개발은 기획/설계->상세계획->구현/테스트->리뷰 순서로 개발을 진행하라.**

- 기획/설계 : superpowers/brainstorming 스킬, Fable 모델 사용
- 상세개발계획 : superpowers/writing-plans 스킬, Fable 모델 사용
- 구현/테스트 : superpowers/subagent-driven-development, superpowers/test-driven-development 스킬, Sonnet 5 모델 사용
  - (SDD 진행 중 태스크별 리뷰는 구현 단계에 속하므로 Sonnet 5로 진행)
- 최종리뷰 : superpowers/requesting-code-review 스킬로 전체 브랜치를 1회 리뷰, Opus 모델 사용
- 실행 시 현재 기본 모델과 다를 시: 상세개발계획·최종리뷰는 해당 모델의 서브 에이전트로 위임하고, 기획/설계(대화형)는 사용자에게 `/model` 전환을 요청한다.
